總覽：模組職責

此模組負責管理「外部資料來源供應商」（Provider1、Provider2、Provider3 等）的同步行為。 系統可在不重啟的情況下，依據 SystemSetting 動態切換目前使用的供應商， 並分別管理「一般資料（general data）」與「報價資料（price data）」兩條獨立的 provider。

類別關係圖

① Singleton — ApiDataProviderManager

採用 eager initialization（餓漢式）單例：類別載入時即建立唯一實例，搭配 private static final 保證執行緒安全的初始化。實例內以兩把 ReentrantLock 分別保護一般資料與報價資料的 provider 切換。

public class ApiDataProviderManager {

    // 餓漢式：類別載入即建立，JVM 保證執行緒安全
    private static final ApiDataProviderManager instance = new ApiDataProviderManager();

    private ApiDataProvider apiDataProvider;        // 一般資料策略
    private ApiDataProvider apiPriceDataProvider;    // 報價資料策略

    private ReentrantLock lock = new ReentrantLock();
    private ReentrantLock priceDataProviderLock = new ReentrantLock();

    public static ApiDataProviderManager getInstance() {
        return instance;
    }
}
為何適用：provider 狀態（當前選用哪家、是否正在切換）必須全系統一致，單例避免多份狀態造成不一致。eager 模式在此情境下簡單可靠，因為這個物件啟動後一定會用到。

② Strategy — ApiDataProvider

ApiDataProvider 抽象類別即為「策略介面」，定義所有供應商必須實作的同步動作。Provider1 / Provider2 / Provider3 為可互換的具體策略。Manager 只持有抽象型別參考，呼叫時不需知道實際是哪一家。

public abstract class ApiDataProvider {
    public abstract void init();
    public abstract void syncCompetition();
    public abstract void syncEvent();
    public abstract void syncMarketInstrument();
    public abstract List<String> syncMarketDetails(long eventId, String[] marketIdArray);
}
熱切換的精髓：更新時先 new 出新策略並完成 init()，最後一行才把欄位指向新物件 — 切換前舊策略持續服務，切換瞬間原子性替換參考，達到「無中斷切換供應商」。

③ Enum Factory — ApiDataProviderType

這是整個模組最精巧之處：以 列舉常數特定方法本體（constant-specific method bodies）把「型別、預設幣別、URL、是否停用、是否為報價源、以及對應的 provider 工廠」全部內聚到每個常數內。等同把多個 switch 分支收斂成型別安全的列舉，新增供應商只需新增一個常數。

public enum ApiDataProviderType {

    PROVIDER1(1) {
        // 每個常數持有自己的策略實例（等同 Factory 產物快取）
        private ApiDataProvider apiDataProvider = new Provider1ApiDataProvider();
        public ApiDataProvider getApiDataProvider() { return apiDataProvider; }
        public CurrencyType getDefaultCurrencyType() { return Setting.ENABLE_PROVIDER1_EXCHANGE ? GBP : EUR; }
        public boolean isDisabled() { return false; }
        public boolean isPriceProvider() { return true; }
    },
    Provider2(2) { /* ... */ },
    Provider3(3) { /* isDisabled() = true，尚未接好 */ };

    private int value;
    public static final ApiDataProviderType[] VALUES = ApiDataProviderType.values();

    // 註冊表查找：以 DB 設定值反查常數，回傳 Optional 避免 null
    public static Optional<ApiDataProviderType> getInstanceOf(int value) {
        for (ApiDataProviderType e : VALUES)
            if (e.value == value) return Optional.of(e);
        return Optional.empty();
    }

    // 強制每個常數提供以下能力 → 編譯期保證完整性
    public abstract ApiDataProvider getApiDataProvider();
    public abstract CurrencyType getDefaultCurrencyType();
    public abstract String getUrl();
    public abstract boolean isDisabled();
    public abstract boolean isPriceProvider();
}
角色由誰扮演
Factory MethodgetApiDataProvider() — 各常數回傳對應具體策略
Registry LookupVALUES + getInstanceOf(int) — 以整數鍵反查
Type-safe Configunique() 整數值對應 DB 的 SystemSetting 值
Feature FlagisDisabled() / isPriceProvider() — 常數層級開關
等同的 GoF 對應：這是 Joshua Bloch《Effective Java》Item 34「以 enum 取代 int 常數 + 用常數特定方法實作行為」的典型範例，同時兼具 Factory Method 與 Strategy 選擇器的角色。

④ Facade — Manager 對外介面

Manager 把「選哪個 provider、加鎖切換、委派呼叫」的複雜度包起來，對外只暴露語意清楚的同步動作。注意一般資料與報價資料委派到不同的 provider 欄位。

public void syncCompetition() { apiDataProvider.syncCompetition(); }       // 一般資料源
public void syncEvent() { apiDataProvider.syncEvent(); }
public void syncMarketInstrument() { apiDataProvider.syncMarketInstrument(); }

public List<String> syncMarketDetails(long eventId, String[] ids) {
    return apiPriceDataProvider.syncMarketDetails(eventId, ids);     // 報價資料源
}

// 對外的熱切換入口（依 updateType 切換不同 provider）
public void updateProviderByHttpRequest(int updateType) {
    if (updateType == 1) updateApiDataProvider();      // 一般
    else                 updateApiPriceDataProvider();  // 報價
}

優點與可改進處

設計優點

  • 開放封閉原則（OCP）：新增供應商只需新增一個列舉常數與一個 ApiDataProvider 子類，無須改動 Manager。
  • 型別安全：以 enum 取代魔術數字 switch，編譯期即保證每個常數實作齊全的能力集。
  • 無中斷熱切換：先建構並 init() 新策略，最後才替換參考，搭配 ReentrantLock 保證切換的原子性。
  • 關注點分離：一般資料與報價資料以兩條 provider + 兩把鎖獨立管理，互不干擾。
  • Optional 防呆：getInstanceOf 回傳 Optional 而非 null，呼叫端被迫面對「找不到」的情況。

可改進處

volatile 缺失：apiDataProvider / apiPriceDataProvider 欄位非 volatile。寫入端在鎖內，但讀取端（syncCompetition() 等）未加鎖也未 volatile，在多執行緒下可能讀到舊參考的可見性問題。建議將兩欄位宣告為 volatile。
Optional 的脆弱鏈：getInstanceOf(...).get() 直接呼叫 .get()，若 DB 設定值非法（查無對應常數）會拋 NoSuchElementException，錯誤訊息不友善。建議改用 .orElseThrow(() -> new Exception("Unknown provider value: " + v))。
強制型別檢查未利用：enum 已提供 isPriceProvider()，但 updateApiPriceDataProvider() 未驗證所選型別確實為報價源，可加上守門檢查避免設定錯置。
updateType 魔術數字：updateProviderByHttpRequest(int updateType) 用 1/else 區分，語意不明確，可改為列舉或具名常數提升可讀性。
